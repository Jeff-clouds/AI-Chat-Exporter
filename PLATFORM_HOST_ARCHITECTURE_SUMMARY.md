# AI 对话平台寄生架构简明地图

> 版本：2026-07-16
> 范围：只描述宿主平台及其对寄生开发的通用约束，不涉及任何现有项目实现

## 1. 先建立一个共同模型

这七个平台本质上都是单页 Web 应用。用户看到的“一个聊天页面”，通常由五层组成：

~~~mermaid
flowchart TD
    Route["会话路由<br/>决定当前是哪一个会话"]
    Shell["应用外壳<br/>侧栏 / 顶栏 / 输入区"]
    Conversation["会话区域<br/>问题与回答的时间序列"]
    Message["消息或轮次<br/>user / assistant / thinking / tool"]
    Segment["内容分段<br/>Markdown / 代码 / 引用 / 卡片 / 图片"]

    Route --> Shell
    Shell --> Conversation
    Conversation --> Message
    Message --> Segment
~~~

寄生型插件最容易犯的错误，是把某一时刻的 DOM 当成完整会话数据库。实际上：

- 路由决定“当前是谁”。
- DOM 只代表“现在挂载了什么”。
- 流式生成会持续修改同一个回答。
- thinking、工具过程和最终回答可能是不同分段，也可能共用一个回答容器。
- 长会话可能回收屏幕外节点。

## 2. 七个平台总览

| 平台 | 会话路由形态 | 消息组织方式 | 长会话加载 | 最重要的寄生边界 |
|---|---|---|---|---|
| ChatGPT | `/c/{conversationId}`，也可能嵌在项目或 GPT 路径内 | turn -> 一个或多个 message -> Markdown/工具内容 | 明确存在窗口化挂载 | 完整会话数据与当前渲染 DOM 是两个层次 |
| 豆包 | `/chat/{conversationId}` | message feed，消息拥有独立身份，回答内部再分 Markdown 段 | 明确存在虚拟滚动 | 不能通过后台自动滚动假装获得完整历史 |
| DeepSeek | `/a/chat/s/{conversationId}` | 用户消息与 AI 回答交替；AI 回答包含思考和最终正文 | 页面存在虚拟列表结构 | thinking 与最终回答属于同一轮的不同内容面 |
| 腾讯元宝 | `/chat/{agentId}/{conversationId}` | human item / AI item；AI 内含推理、正文、引用和卡片 | 当前实测样本完整挂载，超长会话未知 | agent 身份和 conversation 身份都属于路由边界 |
| Gemini | `/app/{conversationId}` | `USER-QUERY` 与 `MODEL-RESPONSE` 组成轮次，正文在结构化内容组件中 | 超长会话行为待验证 | 页面采用组件化 custom elements，旧 class 外壳不可靠 |
| Grok | `/c/{conversationId}`，可能带附加 query | 用户消息、助手消息和 Markdown 内容流 | 超长会话行为待验证 | Cloudflare 是入口层，不是会话架构；宿主页本身客户端负载较重 |
| Kimi | `/chat/{conversationId}` | user segment / assistant segment；一个回答可包含多个 Markdown 分段 | 超长会话行为待验证 | 回答 segment 才是轮次，Markdown block 只是轮次内部片段 |

## 3. ChatGPT

### 宿主结构

ChatGPT 是“会话数据树 + 当前页面窗口”的双层结构。

- URL 中的 conversationId 定义当前会话。
- 会话数据包含消息映射、父子关系和当前分支。
- 页面只渲染当前需要展示的 turn 窗口。
- 一个 assistant turn 可能包含过程消息、工具消息和最终回答等多个 message。
- Markdown 标题、代码块和交互组件属于渲染层。

### 寄生边界

不能仅凭当前页面上的 turn 数量判断完整历史。完整文本、分支顺序和页面可定位节点应被视为不同来源。

## 4. 豆包

### 宿主结构

豆包更接近虚拟化的消息 feed。

- URL 中的数字 ID 定义当前会话。
- 页面根据滚动位置挂载一部分消息。
- 屏幕外消息可能被回收，之后重新挂载。
- 回答正文由多个 Markdown 或渲染片段组成。
- 流式回答会持续修改当前消息内容。

### 寄生边界

DOM 完整性取决于用户实际浏览过哪些窗口。插件不应为了采集历史而擅自改变用户滚动位置。

## 5. DeepSeek

### 宿主结构

DeepSeek 是“虚拟消息列表 + 复合 AI 回答”。

- 会话由 `/a/chat/s/{id}` 标识。
- 用户消息和 AI 回答按时间顺序进入虚拟列表。
- 一轮 AI 回答内部通常同时包含 thinking 和最终正文。
- thinking 与最终正文都可能使用 Markdown 渲染，但语义不同。
- 代码、搜索结果和普通正文是回答内部的内容类型。

### 寄生边界

应先识别一轮消息，再区分这轮内部的 thinking、正文和工具结果。不能把每个 Markdown 容器都当成独立回答。

## 6. 腾讯元宝

### 宿主结构

元宝是“智能体上下文 + 会话 + 问答列表”的三层结构。

- 路由同时携带 agentId 和 conversationId。
- 会话列表中 human item 与 AI item 交替出现。
- AI item 内部可以同时包含推理过程、最终正文、引用、代码和卡片。
- 当前现场中，旧 reasoner 组件仍与新组件体系并存。
- 当前 16 轮样本全部挂载在一个长列表中；更长会话是否回收节点仍需单独观察。

### 寄生边界

agent 切换和 conversation 切换是两个不同状态变化。thinking、正文与引用需要在同一 AI item 内分层理解，不能按相邻 Markdown 数组配对。

## 7. Gemini

### 宿主结构

Gemini 当前是高度组件化的 Angular 页面。

- `/app/{id}` 标识会话。
- `USER-QUERY` 表示用户轮次。
- `MODEL-RESPONSE` / `RESPONSE-CONTAINER` 表示模型轮次。
- `MESSAGE-CONTENT` 承载回答正文。
- Markdown、来源、思考过程和操作按钮位于模型回答组件内部。
- 多草稿、重新生成和分支回答可能改变“一问一答”的简单模型。

### 寄生边界

应以组件语义和轮次关系理解页面，不应长期依赖某一代内部 class。还需要明确产品要读取当前草稿、选中草稿还是全部草稿。

## 8. Grok

### 宿主结构

Grok 是客户端渲染的会话页面，外部还有 Cloudflare 访问保护层。

- `/c/{id}` 定义会话，query 可能携带额外进入上下文。
- Cloudflare 只负责访问验证，不属于消息模型。
- 真实会话页包含用户消息、助手消息和 Markdown 内容。
- 当前真实页面可正常打开，但自动化结构读取负载较高。
- 长会话虚拟化、流式节点复用和分支回答仍需要更轻量的现场观察。

### 寄生边界

必须先区分“被安全验证拦截”和“进入宿主会话后读取失败”。两者是入口可达性与页面架构两个不同问题。

## 9. Kimi

### 宿主结构

Kimi 是“会话滚动区 + 消息 segment + 内容 block”的分层结构。

- `/chat/{id}` 定义会话，query 记录进入方式等附加状态。
- 用户问题占一个 segment。
- AI 回答占一个 assistant segment。
- 同一个回答 segment 内可以出现多个 Markdown block，例如引导段、搜索过程和最终正文。
- 标题、代码和引用属于回答内部内容，不应提升成新的回答轮次。

### 寄生边界

轮次边界应以 segment 为准，内容提取再进入 segment 内部。直接统计 Markdown block 会放大回答数量。

## 10. 进入开发前的六个问题

面对任何平台变化，只回答下面六个问题：

1. 当前会话由 URL 的哪一部分标识？
2. 页面中的一轮消息由什么容器表示？
3. thinking、工具过程和最终正文是同一轮还是不同轮？
4. 回答内部有哪些内容分段？
5. 滚动时旧消息会保留、隐藏还是回收？
6. 流式生成是在修改原节点，还是不断追加新节点？

这六个问题回答清楚后，再讨论插件如何寄生。顺序不能反过来。

## 11. 从平台现实推导开发策略

| 观察到的平台现实 | 应采取的策略 | 禁止做法 |
|---|---|---|
| URL 明确包含会话 ID | 把规范化会话 ID 作为数据隔离键 | 只根据页面标题或问题文本区分会话 |
| 当前 DOM 只是滚动窗口 | 使用平台数据源或按稳定消息身份累计 | 滚动一次后把 DOM 当完整历史 |
| 一轮回答包含 thinking、工具和正文 | 先确定 turn/message，再拆内部内容层 | 把每个 Markdown block 当成一条回答 |
| 流式过程持续修改同一节点 | 按稳定身份更新原记录 | 每次 DOM mutation 都新增一条回答 |
| 页面存在重新生成或多草稿 | 先定义“当前分支 / 当前草稿 / 全部草稿”的产品语义 | 默认按 DOM 顺序把所有回答串在一起 |
| SPA 切换会替换会话根节点 | 路由变化时重建会话级状态和观察范围 | 让旧 observer、旧请求和旧缓存继续工作 |
| 找不到稳定消息 ID | 使用会话内结构关系和短生命周期身份 | 用文本 hash 充当长期跨会话主键 |
| 平台只有当前 DOM，没有历史数据源 | 明确输出范围是“当前挂载 / 已观察” | 在产品上声称完整会话 |

## 12. 七个平台的一句话适配方向

### ChatGPT

以会话数据和当前分支确定完整顺序，以 DOM 负责真实标题和页面定位；不要试图只扫描页面恢复全历史。

### 豆包

以稳定消息身份被动累计用户浏览过的虚拟窗口；不得为了采集完整性自动滚动页面。

### DeepSeek

先以虚拟列表中的真实消息轮次建立 user / assistant 边界，再把 assistant 内的 thinking、搜索和最终正文分层。

### 腾讯元宝

会话身份必须同时考虑 agent 与 conversation；每个 AI item 内分别处理推理、最终正文、引用和卡片。

### Gemini

以 `USER-QUERY` 与 `MODEL-RESPONSE` 的组件关系建立轮次，并先定义多草稿 / 重新生成的产品语义。

### Grok

把 Cloudflare 可达性与会话 DOM 分开诊断；进入真实会话后，再确认消息身份、流式复用和长会话挂载策略。

### Kimi

以 user / assistant segment 确定轮次，再在 assistant segment 内合并多个 Markdown、搜索和引用分段。

## 13. 开发前置门禁

任何平台适配开始前，至少留下以下证据：

~~~text
[ ] 一个真实会话 URL，以及规范化后的会话身份
[ ] 至少两轮问答的容器层级，不只是一组 selector 计数
[ ] thinking / 工具过程 / 最终正文的归属关系
[ ] 回答内部 Markdown / 代码 / 引用 / 卡片的分段关系
[ ] 流式开始、进行中、结束时，同一轮节点是否复用
[ ] 滚动前后消息节点是保留、隐藏还是回收
[ ] A -> B 切换时，旧页面节点和旧异步结果是否短暂残留
[ ] 当前证据能保证完整历史、当前挂载，还是仅已观察范围
~~~

以下任意情况出现时，不允许直接进入实现：

- 只知道 question / answer selector，不知道一轮消息的父容器。
- 只验证了一问一答，没有观察流式和多轮。
- question 与 answer 数量相等，就默认它们一一对应。
- selector 返回非空，但没有检查命中节点的角色。
- 没有区分 thinking、工具过程和最终正文。
- 不知道宿主是否回收屏幕外 DOM。
- 不知道会话切换后旧数据如何失效。

## 14. 交付验收的最小闭环

一项平台适配只有同时满足下面四层，才算完成：

1. 身份正确：A、B 会话不会串数据。
2. 轮次正确：问题、回答、thinking 和工具内容归属正确。
3. 范围诚实：明确是完整会话、当前挂载还是已观察范围。
4. 生命周期正确：流式、滚动、切会话以及寄生功能停止后，相关状态都能收敛。

如果只能证明 selector 命中，最多算“发现了候选 DOM”，不能算平台适配完成。
